# TFT Data Architecture

## Source Priority

The backend TFT data chain is incremental and ordered:

1. `season-pack`
2. `remote` QQ snapshot
3. `cache`
4. `fallback`

`src-backend/data/TftDataProvider.ts` owns that priority. A higher layer may enrich a loaded snapshot, but it should not change the fallback order.

## Season-Pack Schema

The local season-pack adapter is intentionally compatible with JinChanChanTool main-branch data.

Supported entry roots:

- `Resources/`
- `Resources/HeroDatas/`
- a direct season directory such as `Resources/HeroDatas/英雄联盟传奇/`

Required season files:

- `HeroData.json`
- `Equipment.json`

Optional season files:

- `EquipmentData.json`
- `LineUps.json`
- `RecommendedLineUps.json`
- `images/`
- `EquipmentImages/`
- season-level `CorrectionsList.json`

Optional root files:

- `CorrectionsList.json`

The importer tolerates:

- empty `Profession` and `Peculiarity` arrays
- missing `SyntheticPathway`
- tuple-style lineup positions such as `{ "Item1": 1, "Item2": 2 }`
- incomplete sibling seasons, as long as one season has the required files

## Ownership Boundaries

### Importer

`src-backend/data/JinChanSeasonPackLoader.ts` only owns JSON parsing and snapshot mapping:

- champions
- items
- traits
- lineups
- OCR corrections

It does not copy images, generate manifests, or rewrite `public/lineups`.

### Image Syncer

`src-backend/data/JinChanSeasonPackAssetSyncer.ts` owns non-destructive asset staging for season packs:

- reads `images/` and `EquipmentImages/`
- copies only missing files into `public/resources/season-packs/<season>/`
- never deletes or overwrites existing assets
- emits `asset-map.json`
- emits `metadata.json`
- attaches local asset paths back onto snapshot `imageUrl` fields

### OCR Corrections

`src-backend/tft/recognition/OcrCorrectionService.ts` only handles OCR text corrections.

Current runtime contexts:

- `stage`
- `shop`
- `equipment`
- `all`

This layer is for OCR mistakes and typo-like replacements. It is not the home for canonical data aliases.

### Canonical Name Normalization

`src-backend/data/TftNameNormalizer.ts` owns canonical equivalence for data/lookups, such as:

- `反曲弓 -> 反曲之弓`

Use it at comparison and lookup boundaries, not inside template matching itself.

## Field Roles

### Recognition Fields

These fields directly support OCR, template matching, or fixture validation:

- champion names
- item names
- OCR corrections
- local synced image paths

### Strategy Fields

These fields support planning and runtime decisions:

- lineup champions
- core champions
- recommended items
- trait lists

`src-backend/lineup/LineupLoader.ts` remains the automation-specific loader for staged lineup configs under `public/lineups`.

### Display Fields

These fields are safe for UI or reporting layers:

- `imageUrl`
- lineup `quality`
- lineup `notes`
- `updatedAt`

## Conflict Rules

- If a season pack exists, it wins over remote, cache, and fallback data.
- If an asset already exists in the staged season-pack directory, the syncer leaves it unchanged and records it as `existing`.
- If a source image is missing, the syncer records `missing` and the snapshot keeps any previous `imageUrl` fallback.
- If a canonical-name alias differs from fixture or external shorthand, comparisons normalize first and only then decide pass/fail.

## Metadata

Because JinChanChanTool does not ship a stable `version.json`, local metadata is derived rather than required.

`metadata.json` generated by the asset syncer records:

- season name
- source root and season directory
- snapshot versions
- generated timestamp
- latest lineup update seen in imported lineups
- copied/existing/missing asset counts

This metadata is advisory. Loading a season pack must not fail just because metadata is absent.

## Future Merge Boundary

There is still intentional duplication between:

- snapshot lineups loaded through `TftDataProvider`
- staged automation lineups loaded through `LineupLoader`

That duplication is documented here but not removed in this round. The next safe merge step would be a shared lineup projection layer, not a direct replacement of either loader.
