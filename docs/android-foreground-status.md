### Fixture Scaffolding
- Script: `scripts/scaffold-android-foreground-fixture.ts`
- Purpose: generate `android-foreground-fixture.v1` JSON from captured screenshots for one target state.
- Example:
- `node --import tsx scripts/scaffold-android-foreground-fixture.ts --output temp/lobby.json --id android-na-lobby-capture --label "安卓大厅采集" --description "待补录真实大厅截图" --state LOBBY --screenshot path/to/lobby-1.png --screenshot path/to/lobby-2.png`
- Review the generated `expectedDecisionKind` values before treating the fixture as authoritative.

### Real Capture Ingestion
- Script: `scripts/ingest-android-foreground-captures.ts`
- Purpose: generate a real-capture draft fixture, `android-foreground-manifest.v1`, and review crops in one run.
- Example:
- `node --import tsx scripts/ingest-android-foreground-captures.ts --output-dir temp/android-na-lobby --id android-na-lobby-draft --label "安卓大厅真实草稿" --description "真实截图接入草稿" --state LOBBY --screenshot path/to/lobby-1.png --screenshot path/to/lobby-2.png`
- Output:
- `<output-dir>/<id>.json` foreground fixture draft
- `<output-dir>/manifest.json` capture manifest with classified vs expected state
- `<output-dir>/crops/*.png` review crops for anchor and action-point calibration

### Replay Diff
- Script: `scripts/compare-android-foreground-fixtures.ts`
- Purpose: compare two fixture replays frame-by-frame and surface state / verification / decision changes.
- Example:
- `node --import tsx scripts/compare-android-foreground-fixtures.ts --baseline examples/android-foreground-replay/android-na-frontend-synthetic-flow.json --candidate temp/android-na-lobby/android-na-lobby-draft.json`
- Exit code is non-zero when diffs exist, so it can be used in review workflows.

## Known Boundaries And False-Positive Guards

- Non-game desktop windows must classify as `UNKNOWN`, not `LIVE_CONTENT`.
- BlueStacks shell/launcher must classify as `BLUESTACKS_BOOT`, not lobby or live HUD.
- Update/login frontend must not be mistaken for live HUD.
- In-game augment overlays must still classify as `LIVE_CONTENT`.
- Real ingested lobby/queue/accept screenshots can live as `REAL_CAPTURE_DRAFT` before they are curated into the verified real replay path.

## Minimum Real Capture Checklist

Capture all screenshots from the same emulator profile and resolution used for smoke testing.

### Required States
- Lobby: 3 screenshots
- Queue/matchmaking: 3 screenshots
- Ready-check/accept: 3 screenshots
- In-game transition/loading after accept: 3 screenshots

### Required Resolution Rules
- Keep one fixed emulator resolution per capture batch.
- Do not crop the screenshots manually; capture full emulator content area.
- Keep window chrome consistent if using the same smoke window path.

### Required Metadata Per Batch
- Region/server (`NA`)
- Emulator name/version
- Window width x height
- Whether the frame is pre-click, stable, or post-click
- Whether the CTA/button is actually clickable in that frame

### Useful Crop Targets To Auto-Export Later
- Start queue button region
- Cancel queue button region
- Ready-check accept button region
- Any unique lobby anchors near the top or side rails
- Any loading/transition anchor that exists before live HUD appears
