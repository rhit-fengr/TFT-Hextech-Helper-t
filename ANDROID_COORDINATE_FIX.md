# 🎯 安卓模拟器坐标系统彻底修复

## 问题诊断

用户说得完全对！问题根本不在 OCR 识别本身，而在 **坐标定义**。

### 根本原因
1. `TFTProtocol.ts` 中的所有坐标都是**绝对像素值**（基于 1024x768 标准分辨率）
2. 当窗口尺寸不同（如 971x473 安卓模拟器）时，这些坐标就完全错位
3. 例如：`gameStageDisplayNormal = { x: 374, y: 6 }` 只对 1024x768 有效

### 为什么之前识别成 "1-86", "6-6" 等垃圾值
- OCR 捕获的是错误位置的区域
- "1-86" 说明了捕获到了多行文字（包括回合）
- 即使用 4 次确认机制，也只是确认了一个错误的值

## ✅ 已完成的修复

### 1. 改进 TFTProtocol.ts 中的阶段识别区域
**改为百分比坐标形式**：
```typescript
// 修改前（绝对坐标，仅对 1024x768 有效）
gameStageDisplayNormal = {
    leftTop: { x: 374, y: 6 },
    rightBottom: { x: 403, y: 22 }
}

// 修改后（百分比坐标，自动适配任何窗口尺寸）
gameStageDisplayNormal = {
    leftTop: { x: 0.25, y: 0.01 },      // 占窗口宽度的 25%, 高度的 1%
    rightBottom: { x: 0.42, y: 0.035 }  // 占窗口宽度的 42%, 高度的 3.5%
}
```

**改进的其他阶段区域**：
- `gameStageDisplayStageOne`: 第一阶段的阶段显示
- `gameStageDisplayShopOpen`: 商店打开时的位置调整
- `gameStageDisplayTheClockworkTrails`: 发条鸟模式

### 2. 改进 ScreenCapture.ts 中的坐标转换逻辑
**新增智能坐标检测**：
- 自动识别**百分比坐标**（0-1 范围） vs **绝对坐标**（1-1024）
- 百分比坐标：用 `actualWidth * percentage` 计算
- 绝对坐标：保留向后兼容性（乘以 scaleX/scaleY）

**涉及的方法**：
- `toAbsoluteRegion()`: 将相对区域转换为绝对屏幕坐标
- `toAbsolutePoint()`: 将相对坐标点转换为绝对坐标点

### 3. 改进 TftOperator.ts 的文本提取逻辑
**修正 `extractLikelyStageText()` 方法**：
- 当所有合理性检查都失败时，返回 **空字符串** 而非垃圾值
- 避免采用 "1-86", "6-6" 等不合理的阶段

## 📊 技术细节

### 百分比坐标的计算方式
当窗口是 971x473 时：
- `leftTop.x = 0.25` → 实际 x = 54（窗口左边界） + 0.25 * 971 = 295 像素
- `leftTop.y = 0.01` → 实际 y = 110（窗口上边界） + 0.01 * 473 = 115 像素

### 向后兼容性
- 旧的绝对坐标仍然能用（如 `{ x: 374, y: 6 }` 的其他区域）
- 系统能自动判断是百分比（< 1）还是绝对值（> 10）
- 不需要一次性改所有坐标
